## 1. COBOL + CGI Integration

**Risk: Medium**

### The Challenge

GnuCOBOL compiles to C, then to a native binary. Making it work as a CGI program requires:
- Reading environment variables (`CONTENT_LENGTH`, `QUERY_STRING`, `REQUEST_METHOD`)
- Reading POST data from standard input
- Writing HTTP response headers to standard output
- Writing HTML body to standard output

COBOL was not designed for web programming. There is no built-in HTTP library.

### Mitigation Strategy

GnuCOBOL supports two key features that make this work:

```cobol
* Read environment variable
ACCEPT WS-CONTENT-LENGTH FROM ENVIRONMENT "CONTENT_LENGTH"

* Write to stdout (CGI output)
DISPLAY "Content-Type: text/html"
DISPLAY ""
DISPLAY "<h1>Hello from COBOL</h1>"
```

**Action plan:**
- Day 1-2: Build a "Hello World" COBOL CGI program
- Verify it compiles, deploys to Apache, and returns HTML
- If it works → the rest of the project is de-risked
- If it fails → investigate GnuCOBOL CGI examples from the community (multiple exist)

### Fallback

If CGI proves problematic, use a thin Python/Bash wrapper that invokes the COBOL binary and pipes the output to Apache. This adds one dependency but preserves the COBOL calculation engine.

## 2. Form Data Parsing in COBOL

**Risk: Medium**

### The Challenge

POST data arrives URL-encoded: `client=Dupont%20SARL&amount=1500.00`

COBOL has no built-in URL decoder. Parsing requires:
- Splitting the string on `&` (field separator)
- Splitting each field on `=` (key-value separator)
- Decoding `%XX` hex sequences (e.g., `%20` → space)
- Handling `+` as space

### Mitigation Strategy

Write a reusable `cgi-utils.cob` module (~50-80 lines of COBOL) that:
1. Takes the raw POST string as input
2. Splits into key-value pairs using `INSPECT` and `UNSTRING`
3. Decodes percent-encoded characters
4. Returns a structured record with named fields

```cobol
UNSTRING WS-POST-DATA DELIMITED BY "&"
    INTO WS-PAIR(1) WS-PAIR(2) WS-PAIR(3)
    TALLYING IN WS-PAIR-COUNT.
```

This module is written once and reused by all CGI programs.

### Fallback

Limit form field names to simple ASCII (no special characters needed for field names). Client names with special characters can be handled by restricting input or using a simplified encoding.

## 3. PostScript Learning Curve

**Risk: Low**

### The Challenge

PostScript is a stack-based language with its own syntax and coordinate system. It's different from any mainstream language.

### Mitigation Strategy

PostScript for document layout is actually straightforward. The commands needed for an invoice are limited:

| Command | Purpose |
|---------|---------|
| `moveto` | Position the cursor |
| `show` | Draw text at current position |
| `lineto` + `stroke` | Draw a line |
| `findfont` + `scalefont` + `setfont` | Set typeface and size |
| `setgray` | Set color (grayscale) |
| `showpage` | Emit the page |

An invoice template uses ~15 distinct PostScript commands. The full template is 100-150 lines.

### Resources

- *PostScript Language Tutorial and Cookbook* (Adobe "Blue Book") — freely available
- Ghostscript documentation
- The template can be developed incrementally: start with text, add lines, add the table

## 4. COBOL Calling Ghostscript

**Risk: Low**

### The Challenge

After generating the `.ps` file, COBOL must invoke Ghostscript to convert it to PDF.

### Mitigation Strategy

GnuCOBOL supports shell command execution:

```cobol
MOVE "gs -sDEVICE=pdfwrite -dNOPAUSE -dBATCH -dQUIET -o "
    TO WS-GS-CMD.
STRING WS-GS-CMD DELIMITED SPACE
       WS-PDF-PATH DELIMITED SPACE
       WS-PS-PATH DELIMITED SIZE
    INTO WS-FULL-CMD.
CALL "SYSTEM" USING WS-FULL-CMD.
```

**Important security note:** The file paths passed to Ghostscript are generated by the system (e.g., `INV-2026-0042.ps`), never from user input. This eliminates shell injection risk.

## 5. Session Management Without a Framework

**Risk: Medium**

### The Challenge

CGI is stateless. Every request spawns a new process. There is no built-in concept of sessions, cookies, or authentication.

### Mitigation Strategy

Implement a minimal session system:

1. **Login:** User submits username/password → COBOL verifies against stored hash → generates a random session token → stores in `sessions.dat` (ISAM) → sets `Set-Cookie: COBILL_SID=token`
2. **Authentication:** On each request, COBOL reads `HTTP_COOKIE` env var → extracts token → looks up in `sessions.dat` → if valid and not expired, proceed; else redirect to login
3. **Logout:** Delete session record from `sessions.dat` → clear cookie

```cobol
ACCEPT WS-COOKIE FROM ENVIRONMENT "HTTP_COOKIE".
* Parse COBILL_SID from cookie string
PERFORM EXTRACT-SESSION-TOKEN.
* Look up in sessions.dat
READ SESSION-FILE KEY IS WS-SESSION-TOKEN
    INVALID KEY
        PERFORM REDIRECT-TO-LOGIN
    NOT INVALID KEY
        PERFORM CHECK-SESSION-EXPIRY
END-READ.
```

### Fallback

For MVP, skip authentication entirely (single-user local app). Add it in Week 3 as a core feature.

## 6. Development Environment

**Risk: Low**

### The Challenge

Development machine runs macOS. GnuCOBOL and Apache CGI testing are easier on Linux.

### Mitigation Strategy

Three options, in order of preference:

1. **Develop directly on the VPS via SSH** — All tools are on the server. Use VS Code Remote SSH extension.
2. **Local Linux VM** — UTM (macOS native) or Multipass for Ubuntu VM.
3. **GnuCOBOL on macOS** — GnuCOBOL can be installed via Homebrew (`brew install gnucobol`). Apache is pre-installed on macOS. Test locally, deploy to Linux.

**Recommendation:** Option 1 (develop on VPS). It eliminates all environment discrepancies and the deployment step becomes trivial.
